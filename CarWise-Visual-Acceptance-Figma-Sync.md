# CarWise 视觉验收清单 + Figma 同步说明

## 1. 文档目的

这份文档用于固定 CarWise 当前移动端视觉版本，避免后续继续凭感觉美化。它同时服务两件事：

- 代码侧：判断当前页面是否达到「高级车主管家风」。
- Figma 侧：按当前项目样式反推 Design Tokens、Components 和 3 张高保真页面。

本轮验收范围只包括：

- 首页。
- 权益页。
- 提车验车页。

购车页、问题页、用车页、档案页暂不重新设计，后续只按这 3 页的规则延展。

## 2. 截图基准

### 2.1 截图尺寸

统一使用手机尺寸：

- 宽度：390 px 或 430 px。
- 高度：844 px。
- 页面缩放：100%。
- 截图范围：首屏为主，必要时补充长截图。

### 2.2 文件命名

建议保存到 `docs/visual-baseline/`。

| 页面 | 文件名 |
| --- | --- |
| 首页 | `home-mobile-baseline.png` |
| 权益页 | `equity-mobile-baseline.png` |
| 提车验车页 | `delivery-mobile-baseline.png` |

### 2.3 截图要求

- 首页截图要看到车辆状态、当前最该处理、关键数据入口。
- 权益页截图要看到落实进度、待处理数量、权益清单第一屏。
- 提车页截图要看到验车进度、关键未完成项、第一组验车项。
- 截图中不能出现乱码。
- 截图中不能出现不适合现场使用的敏感表达，例如「销售承诺」。

## 3. 真实流程检查

### 3.1 检查路径

按以下路径完整走一遍：

1. 从首页进入权益页。
2. 修改一条权益状态。
3. 进入提车验车页。
4. 勾选一个验车项。
5. 标记一个问题。
6. 回到首页。
7. 检查首页待办、验车进度、未解决问题是否变化。

### 3.2 验收标准

| 检查项 | 通过标准 |
| --- | --- |
| 首页状态 | 能看到车辆状态、待办数量、验车进度、权益数量。 |
| 权益状态修改 | 修改后状态标签立即变化。 |
| 验车勾选 | 勾选后该项变为完成态，进度数字变化。 |
| 问题标记 | 标记后问题入口有风险提示，首页未解决问题数量变化。 |
| 返回首页 | 首页能反映最新状态，不需要用户手动刷新。 |
| 低调模式 | 打开后权益名称和金额不直接暴露。 |
| 中文显示 | 页面没有乱码、旧文案和不合适的现场表达。 |

## 4. Design System 固化

Design System 以 `Visual-Style-Guide.md` 为准，代码和 Figma 都不得另起一套。

### 4.1 Design Tokens

必须同步以下 Token：

- 页面背景。
- 内容底色。
- 主文字、次级文字、弱提示文字。
- 深紫主色。
- 蓝绿色辅助色。
- 成功、提醒、风险状态色。
- 分割线。
- 8 px 圆角。
- 极轻阴影。

### 4.2 Components

Figma 至少建立以下组件：

- 主按钮。
- 次按钮。
- 危险按钮。
- 状态标签。
- 进度条。
- 底部导航。
- 权益卡。
- 验车项。
- 重点待办区。
- 页面标题区。

组件命名建议：

| 组件 | Figma 名称 |
| --- | --- |
| 主按钮 | `Button / Primary` |
| 次按钮 | `Button / Secondary` |
| 危险按钮 | `Button / Danger` |
| 状态标签 | `Status / Pill` |
| 进度条 | `Progress / Linear` |
| 底部导航 | `Navigation / Bottom` |
| 权益卡 | `Card / Equity` |
| 验车项 | `List Item / Inspection` |
| 重点待办区 | `Panel / Priority Task` |
| 页面标题区 | `Header / Page Title` |

## 5. Figma 同步步骤

### 5.1 建立页面

Figma 文件建议包含 2 个页面：

1. `Design System`
2. `MVP Screens`

### 5.2 同步顺序

1. 先建 Design Tokens。
2. 再建基础组件。
3. 按当前代码页面反推首页。
4. 按当前代码页面反推权益页。
5. 按当前代码页面反推提车验车页。
6. 最后补页面之间的点击跳转。

### 5.3 不允许的同步方式

- 不让 Figma 自由生成新风格。
- 不使用汽车海报式背景。
- 不使用大面积渐变。
- 不增加大量装饰图形。
- 不改变低调模式的产品意图。
- 不把权益页重新叫回「销售承诺页」。

## 6. 三张高保真稿要求

### 6.1 首页

必须包含：

- 当前车辆状态。
- 当前最该处理。
- 待确认、未落实权益、验车进度、未解决问题。
- 下一次提醒。
- 最近时间线。
- 快速导入入口。

验收重点：

- 第一眼像车主管家，不像上传工具页。
- 首屏能回答「现在最该处理什么」。
- 信息密度适中，不堆卡片。

### 6.2 权益页

必须包含：

- 权益落实进度。
- 待处理数量。
- 已确认数量。
- 权益清单。
- 状态切换。
- 编辑和删除入口。
- 低调模式。

验收重点：

- 未落实权益能一眼看到。
- 已落实内容不抢视觉。
- 现场使用不尴尬，不暴露过多敏感信息。

### 6.3 提车验车页

必须包含：

- 整体验车进度。
- 关键未完成项。
- 未解决问题。
- 分组验车清单。
- 勾选入口。
- 问题留证入口。
- 交付前总检查。

验收重点：

- 单手可操作。
- 有问题时能快速留证。
- 进度和风险比装饰更突出。

## 7. 进入下一阶段的标准

满足以下条件后，再开始把风格套到购车页、问题页、用车页和档案页：

- 3 张手机截图已保存。
- 真实流程检查通过。
- `Visual-Style-Guide.md` 已和当前页面一致。
- Figma 已有 Design Tokens 和核心组件。
- 首页、权益页、提车页 3 张高保真稿已完成。
- 没有中文乱码。
- 没有旧的「销售承诺」可见表达。
