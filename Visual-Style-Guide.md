# CarWise 移动端视觉规范

## 1. 产品气质

CarWise 第一版定位为「新能源车主私人管家」，不是汽车广告页，也不是社区信息流。视觉目标是让车主在购车、提车和早期用车阶段保持心里有数。

关键词：

- 可靠：重要状态明确，不制造紧张感。
- 清楚：一屏能判断现在最该处理什么。
- 有秩序：报价、权益、验车、问题、费用和提醒分开展示。
- 轻专业：像认真做事的工具，不像复杂后台。
- 克制科技感：有新能源车气质，但不做发光、炫技和海报化效果。

一句话标准：看起来让人放心，用起来让人不慌。

## 2. 颜色

| 用途 | 颜色 | 使用方式 |
| --- | --- | --- |
| 页面背景 | `#F3F5F7` | App 主背景。 |
| 浅背景 | `#F8FAFB` | 页面内的浅层背景。 |
| 内容底色 | `#FFFFFF` | 主要内容承载面。 |
| 次级内容底色 | `#F7F9FB` | 轻量信息块、状态区域。 |
| 主文字 | `#171923` | 标题、关键数字、主要内容。 |
| 次级文字 | `#626D7A` | 说明、日期、备注。 |
| 弱提示文字 | `#8B95A1` | 空状态、占位提示。 |
| 分割线 | `#E3E8EE` | 边框、列表分隔。 |
| 主色深紫 | `#3A245F` | 主按钮、当前导航、关键状态。 |
| 深紫强调 | `#24163C` | 少量强强调背景。 |
| 紫色浅底 | `#F0EBF7` | 当前态、轻强调。 |
| 蓝绿色辅助 | `#0E7F86` | 进度、检查、辅助操作。 |
| 蓝绿色浅底 | `#E8F6F5` | 进度和检查的浅底状态。 |
| 成功 | `#157A48` | 已确认、已落实、已解决。 |
| 提醒 | `#A16000` | 待落实、即将到期、签字前提醒。 |
| 风险 | `#B43A2F` | 有争议、待处理、严重问题。 |

规则：

- 不使用大面积渐变。
- 不使用大面积黑紫背景。
- 紫色只做品牌识别和关键操作。
- 蓝绿色用于进度、验车和轻操作，不抢主色。
- 红、橙、绿只服务状态判断，不做装饰。

## 3. 字体

字体栈：

```css
font-family: Inter, "PingFang SC", "Microsoft YaHei", Arial, sans-serif;
```

| 层级 | 字号 | 字重 | 用途 |
| --- | --- | --- | --- |
| 页面标题 | 26 px | 700-800 | 首页、权益页、提车页主标题。 |
| 区块标题 | 15-17 px | 700 | 重点区域标题。 |
| 关键数字 | 22-36 px | 800 | 待办数量、权益进度、验车进度。 |
| 正文 | 14-15 px | 400-600 | 列表和说明文字。 |
| 辅助文字 | 12-13 px | 400-600 | 日期、状态补充、备注。 |
| 标签文字 | 11-12 px | 700-800 | 状态标签、分类标签。 |

规则：

- 不使用随屏幕宽度变化的字号。
- 不使用负字距。
- 标题不夸张放大，避免营销页气质。
- 卡片和列表内文字保持紧凑、清楚。

## 4. 页面结构

### 首页

目标：打开就知道当前状态和下一件事。

结构：

1. 顶部车辆状态区：车辆、状态、待办数、验车进度、权益数。
2. 低调模式开关。
3. 当前最该处理区域。
4. 关键数据入口。
5. 下一次提醒。
6. 最近时间线。
7. 快速导入入口。

### 权益页

目标：一眼看出哪些权益没落实。

结构：

1. 页面标题和说明。
2. 低调模式开关。
3. 权益落实进度。
4. 待处理和已确认概览。
5. 权益清单。

### 提车验车页

目标：提车现场好用、不慌。

结构：

1. 页面标题和说明。
2. 整体验车进度。
3. 关键未完成项和未解决问题。
4. 分组验车项。
5. 交付前总检查。

## 5. 核心组件

### 5.1 内容承载面

- 圆角：8 px。
- 边框：1 px 浅灰。
- 阴影：极轻，只用于从背景中分离。
- 不做卡片嵌套卡片。
- 不把所有信息都堆成同一种卡片，要区分状态区、待办区、列表区。

### 5.2 按钮

| 类型 | 样式 | 用途 |
| --- | --- | --- |
| 主按钮 | 深紫底、白字 | 保存、处理待确认、关键流程入口。 |
| 次按钮 | 白底、浅边框、深色字 | 编辑、继续验车、进入详情。 |
| 危险按钮 | 风险浅底、风险色文字 | 删除、签字前提醒、风险操作。 |
| 图标按钮 | 小尺寸线性图标 | 勾选、拍照、编辑、删除。 |

规则：

- 一个区域最多放 1 个主按钮。
- 删除按钮不和保存按钮紧挨着。
- 按钮文字要短，不写说明句。

### 5.3 状态标签

| 状态 | 颜色 |
| --- | --- |
| 已确认、已落实、已解决、已完成 | 绿色 |
| 待确认、待落实、未开始、即将到期 | 橙色 |
| 有争议、待处理 | 红色 |
| 已忽略、草稿 | 灰色 |

规则：

- 状态必须有文字，不只靠颜色。
- 同一状态在所有页面保持同一颜色。
- 风险状态不能被弱化到看不见。

### 5.4 进度条

- 背景使用浅灰。
- 完成进度使用蓝绿色。
- 进度条只表达完成度，不做装饰性动画。
- 关键数字和进度条必须同时出现。

### 5.5 底部导航

- 固定在底部。
- 当前页面使用紫色浅底。
- 图标使用线性风格。
- 文案保持短：`首页`、`购车`、`提车`、`用车`、`档案`、`权益`。

### 5.6 权益卡

权益卡包含：

- 权益类型。
- 权益名称。
- 金额或规格。
- 当前状态。
- 编辑、删除。
- 状态切换。

规则：

- 低调模式下隐藏具体名称和金额，只显示「权益事项」和「明细已记录」。
- 未落实和有争议状态要清楚。
- 已落实内容可以弱化，但不能消失。

### 5.7 验车项

验车项包含：

- 勾选入口。
- 检查内容。
- 问题留证入口。

规则：

- 单手可点，触控区域足够大。
- 已完成项弱化，但仍可识别。
- 有问题项使用风险色标记。
- 分组标题必须显示完成数量。

## 6. 禁止项

- 不做汽车广告页。
- 不做大面积渐变。
- 不做发光按钮。
- 不堆叠过多卡片。
- 不用过多阴影。
- 不用社区信息流风格。
- 不在可见界面使用「销售承诺」作为核心表达，统一使用「权益」。

## 7. 套用规则

后续页面按这 3 个重点页面延展：

1. 先复用颜色、字体、按钮、状态标签和底部导航。
2. 再根据页面目标选择布局，不直接复制卡片堆叠。
3. 购车页、问题页、用车页、档案页只做延展，不重新发明视觉语言。
4. Figma 以当前项目样式为准，不自由发挥。
